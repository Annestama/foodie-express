02
Gambaran Umum

Apa itu FoodOrder?

Ekosistem pemesanan makanan real-time yang terdiri dari dua aplikasi terpisah, terhubung melalui shared database.

🛒
Sisi Pelanggan
User App — user_app.py
Aplikasi untuk pelanggan: mendaftar akun, login, memilih restoran, melihat menu, memesan makanan, checkout, dan melacak status pesanan secara real-time (polling 5 detik).
  • Login & Register akun
  • Dashboard katalog restoran + pencarian
  • Halaman menu per restoran
  • Keranjang belanja & checkout
  • Riwayat & tracking pesanan real-time
🏪
Sisi Restoran
Restaurant App — restaurant_app.py
Aplikasi untuk kasir/admin: memilih restoran, mengelola antrian pesanan masuk, mengupdate status pesanan secara sequential, manajemen menu (CRUD), dan melihat laporan penjualan.
  • Login kasir — pilih restoran
  • Dashboard transaksi masuk (polling 10 detik)
  • Update status pesanan berurutan
  • Manajemen menu (Tambah, Edit, Hapus)
  • Laporan pendapatan & menu terlaris
03
Arsitektur

Struktur Proyek & Arsitektur MVC

uas-pbo/ ├── models/ ← MODEL LAYER (OOP) │ ├── base.py [Abstraksi] │ ├── item_menu.py [Pewarisan + Polimorfisme] │ ├── pesanan.py [Enkapsulasi] │ └── keranjang.py [Enkapsulasi] ├── database/ ← DATA ACCESS LAYER │ ├── db_manager.py [SQLite CRUD] │ └── seed_data.py [Data Awal] ├── views/ ← VIEW LAYER (Tkinter GUI) │ ├── user_views.py [6 Frame UI User] │ └── restaurant_views.py [4 Frame UI Admin] ├── user_app.py ← CONTROLLER (User) ├── restaurant_app.py ← CONTROLLER (Admin) └── pemesanan.db [SQLite Database]
Model Layer
Berisi class-class Python murni (POJO) yang mendemonstrasikan 4 pilar OOP. Tidak bergantung pada UI.
View Layer
Komponen Tkinter (Frame) untuk menampilkan antarmuka pengguna. Total 10+ Frame di dua aplikasi.
Controller Layer
Entry point tiap aplikasi. Mengelola navigasi, state session, dan menjembatani Model ↔ View.
Data Access Layer
DatabaseManager mengelola koneksi SQLite, semua operasi CRUD, dengan WAL mode untuk concurrent access.
04
4 Pilar OOP — Pilar #1

Abstraksi (Abstraction)

Membuat blueprint menggunakan Abstract Base Class (ABC) yang mewajibkan subclass mengimplementasikan method tertentu.

📁 models/base.py

from abc import ABC, abstractmethod class Pengguna(ABC): """Abstract Base Class — tidak bisa diinstansiasi langsung.""" def __init__(self, nama: str): self.nama = nama @abstractmethod def tampilkan_dashboard(self): """Kontrak: WAJIB di-override oleh subclass.""" pass def sapa(self) -> str: return f"Selamat datang, {self.nama}!"

Implementasi di Subclass

# user_app.py class Pelanggan(Pengguna): def tampilkan_dashboard(self): self.app_root.deiconify() print(f"Dashboard Pelanggan aktif.") # restaurant_app.py class Kasir(Pengguna): def tampilkan_dashboard(self): self.app_root.deiconify() print(f"Dashboard Kasir aktif.")
💡 Mengapa ini Abstraksi?
Class Pengguna hanya membuat aturan (kontrak) bahwa subclass WAJIB memiliki tampilkan_dashboard(). Class ini tidak bisa di-instansiasi langsung — memaksa developer menggunakan Pelanggan atau Kasir.

4 Pilar OOP — Pilar #2

Enkapsulasi (Encapsulation)

Melindungi data internal menggunakan atribut private (__) dan protected (_). Akses hanya melalui getter/setter yang memiliki validasi.

📁 models/pesanan.py — Atribut Private

class Pesanan: STATUS_FLOW = ["Menunggu", "Dikonfirmasi", "Diproses", "Dikirim", "Selesai"] def __init__(self, ...): # Atribut PRIVATE — dilindungi self.__total_harga = total_harga self.__status_pengiriman = status # GETTER — satu-satunya cara baca def get_total(self) -> float: return self.__total_harga # SETTER — hanya bisa maju 1 langkah def set_status(self, new_status): current_idx = self.STATUS_FLOW.index(...) new_idx = self.STATUS_FLOW.index(new_status) if new_idx == current_idx + 1: self.__status_pengiriman = new_status return True return False

📁 models/keranjang.py — Atribut Protected

class KeranjangBelanja: def __init__(self): # Protected — akses via method saja self._items: list[dict] = [] def tambah_item(self, ...): if qty <= 0: return False # Validasi duplikat & kalkulasi subtotal def get_items(self) -> list: return list(self._items) # Salinan!
💡 Mengapa ini Enkapsulasi?
Data sensitif (__total_harga, __status_pengiriman, _items) tidak bisa dimodifikasi langsung dari luar class. Status pesanan hanya bisa maju 1 langkah melalui set_status() — tidak bisa lompat atau mundur.

4 Pilar OOP — Pilar #3

Pewarisan (Inheritance)

Child class mewarisi atribut dan method dari parent class, lalu menambahkan atribut unik miliknya sendiri.

📁 models/item_menu.py

# PARENT CLASS class ItemMenu: def __init__(self, id, nama, harga, restoran_id): self.id = id self.nama = nama self.harga = harga self.restoran_id = restoran_id # CHILD CLASS — Makanan class Makanan(ItemMenu): def __init__(self, ..., level_pedas="Tidak Pedas"): super().__init__(...) # Warisi dari ItemMenu self.level_pedas = level_pedas # Atribut unik # CHILD CLASS — Minuman class Minuman(ItemMenu): def __init__(self, ..., is_dingin=True): super().__init__(...) # Warisi dari ItemMenu self.is_dingin = is_dingin # Atribut unik

💡 Mengapa ini Pewarisan?
Makanan dan Minuman tidak perlu mendefinisikan ulang atribut id, nama, harga — semua diwarisi dari ItemMenu via super().__init__(). Masing-masing hanya menambah atribut spesifik.

4 Pilar OOP — Pilar #4

Polimorfisme (Polymorphism)

Satu method yang sama menghasilkan perilaku berbeda tergantung objek pemanggilnya (Method Overriding).

📁 models/item_menu.py — Override tampilkan_info()

# Parent method (dasar) class ItemMenu: def tampilkan_info(self) -> str: return f"{self.nama} - Rp{self.harga:,.0f}" # Override di Makanan class Makanan(ItemMenu): def tampilkan_info(self) -> str: return f"[Makanan] {self.nama}" # Override di Minuman class Minuman(ItemMenu): def tampilkan_info(self) -> str: return f"[Minuman] {self.nama}"

Factory Function (Polimorfisme Konstruksi)

def buat_item_dari_db(row: dict): if row['tipe'] == 'makanan': return Makanan(...) # Objek Makanan elif row['tipe'] == 'minuman': return Minuman(...) # Objek Minuman else: return ItemMenu(...) # Objek generik

Contoh Output

[Makanan] Rendang Sapi
[Makanan] Ayam Pop
[Minuman] Es Teh Manis
[Minuman] Matcha Latte Dingin
💡 Mengapa ini Polimorfisme?
Saat UI meloop daftar menu dan memanggil item.tampilkan_info(), hasilnya otomatis berbeda tergantung apakah objeknya Makanan atau Minuman. Nama method persis sama, tapi perilaku berbeda.

Pemanggilan di View

# views/user_views.py — MenuFrame for item in self.menu_items: # Polimorfisme terjadi di sini! lbl = tk.Label(text=item.tampilkan_info())

Database

Desain Database SQLite

Shared database yang digunakan oleh kedua aplikasi, dengan WAL mode untuk concurrent access.

Entity Relationship

📦 users
id | username | password (SHA-256) | nama_lengkap

🏪 restoran
id | nama | deskripsi | aktif

🍕 menu
id | restoran_id (FK) | nama | harga | tipe | level_pedas | is_dingin

📋 pesanan
id | restoran_id (FK) | nama_pemesan | waktu | status | total_harga

📝 detail_pesanan
id | pesanan_id (FK) | menu_id (FK) | nama_menu | qty | subtotal

DatabaseManager Features

Koneksi
WAL mode + Foreign Key ON untuk concurrent read/write antara User App dan Restaurant App.
Auto-Init
Tabel dibuat otomatis saat pertama kali DatabaseManager diinstansiasi. Data demo via seed_data.py.
CRUD Operations
16+ method: get_semua_restoran(), buat_pesanan(), update_status_pesanan(), get_menu_terlaris(), dll.
Security
Password di-hash menggunakan SHA-256 sebelum disimpan. Method _hash_password() bersifat private.

Fitur Teknis

Fitur Teknis Unggulan

🔄
Real-Time Polling
User App polling setiap 5 detik, Restaurant App setiap 10 detik. Menggunakan tkinter.after() untuk non-blocking background refresh.
🔒
Auth & Security
Sistem login/register dengan password hashing SHA-256. Username bersifat UNIQUE constraint di database.
🏭
Factory Pattern
buat_item_dari_db() menerapkan Factory Method — membuat objek Makanan atau Minuman berdasarkan data database.
📊
Laporan & Statistik
Restaurant App menyediakan halaman laporan dengan total pendapatan dan ranking menu terlaris berdasarkan query SQL agregasi.
💾
WAL Mode
SQLite menggunakan Write-Ahead Logging agar kedua aplikasi bisa read/write database secara bersamaan tanpa lock.
🌱
Auto-Seed Data
Database otomatis terisi 3 restoran dan 24 item menu demo saat pertama kali dijalankan. Skip jika sudah ada data.

Teknologi & Deployment

Cara Menjalankan

Step 1 — Jalankan User App
python user_app.py
Buka terminal, arahkan ke folder proyek, lalu jalankan command di atas. Register akun baru, lalu login.
Step 2 — Jalankan Restaurant App
python restaurant_app.py
Buka terminal kedua (bersamaan), pilih restoran, dan kelola pesanan masuk.
💡 Tips Demo
Jalankan kedua aplikasi bersamaan untuk melihat real-time update. Saat kasir mengubah status pesanan, pelanggan akan melihat perubahan dalam 5 detik.
